# Оценка перехода на UUIDv7

В текущей реализации идентификаторы в сервисах в основном создаются через `uuid.New()`, то есть используются случайные UUID версии 4. Такой подход прост и надежен, но при массовых вставках хуже укладывается в индекс по первичному ключу, чем упорядоченные по времени идентификаторы.

Для этого проекта потенциальная польза от `UUIDv7` есть прежде всего в быстро растущих таблицах:

- `posts`
- `photos`
- `signals`
- `signal_photos`
- `comments`
- `notifications`
- `report_votes`
- `trust_events`

Именно в этих сущностях записи добавляются регулярно и в заметном количестве. В таких таблицах `UUIDv7` обычно дает более предсказуемый порядок вставки, снижает фрагментацию индекса и делает последовательные выборки по свежим данным более «дружелюбными» к хранилищу.

При этом немедленный перевод всей базы на `UUIDv7` в рамках текущей итерации не выглядит оправданным по нескольким причинам:

- в проекте уже много внешних связей между таблицами;
- часть данных и связанного кода уже существует в рабочей базе;
- смешанный режим, где часть таблиц использует старые UUID, а часть новые, придется отдельно документировать и тестировать;
- прямой выигрыш будет заметен не во всех таблицах, а в первую очередь только в самых «потоковых».

В итоге наиболее разумным сценарием выглядит не разовая миграция существующих идентификаторов, а отдельный этап, где:

1. выбираются конкретные таблицы с наибольшим потоком вставок;
2. проверяется поддержка генерации `UUIDv7` на уровне используемой библиотеки и окружения;
3. после этого новые записи начинают создаваться уже в формате `UUIDv7`;
4. исторические записи при этом не переписываются без отдельной необходимости.

Вывод по текущему состоянию такой: `UUIDv7` для проекта потенциально полезен, особенно для сигналов, комментариев, уведомлений и журналов событий, но прямо сейчас его лучше рассматривать как отдельную оптимизационную итерацию, а не смешивать с текущими функциональными доработками.
