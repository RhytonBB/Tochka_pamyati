# «Точка памяти»

«Точка памяти» — учебный геопортал, посвященный памятникам и мемориальным объектам. Система позволяет добавлять точки на карту, публиковать посты с фотографиями, отправлять сигналы об угрозах, оставлять комментарии, подавать жалобы и работать с модерацией.

Проект собран как многокомпонентное приложение. Основной сервер написан на Go, клиентская часть — на Vue 3, данные хранятся в PostgreSQL с расширением PostGIS. Отдельно вынесены два сервиса проверки контента: один анализирует текст, второй — изображения.

## Что уже реализовано

- карта объектов культурной памяти;
- добавление новой точки с возможностью сразу приложить первый пост;
- публикация постов к уже существующим точкам;
- загрузка и хранение фотографий;
- сигналы об угрозах и обсуждение сигналов в комментариях;
- жалобы на точки, посты, фотографии, сигналы и комментарии;
- модерация пользовательского контента;
- уведомления, санкции и уровень доверия;
- журнал важных событий системы;
- проверка текста и изображений перед публикацией.

## Архитектура

Проект разделен на четыре самостоятельные части:

1. `project2/front` — клиентская часть.
2. `project2/back` — основной сервер и API.
3. `text-filter` — сервис проверки текста.
4. `image-filter` — сервис проверки изображений.

Схема работы выглядит так:

- браузер обращается к клиентской части;
- клиентская часть отправляет запросы в API;
- сервер сохраняет данные в PostgreSQL и файлы в локальное хранилище;
- при публикации текста и изображений сервер вызывает два внешних сервиса проверки;
- по результатам проверки сервер либо пропускает материал, либо помечает его как рискованный, либо требует подтверждение, либо отправляет на модерацию.

## Сервис проверки текста

Сервис `text-filter` запускается отдельно от основного сервера. Он принимает текст по `HTTP` и возвращает технический результат проверки.

Внутри используются две проверки:

- проверка грубой лексики;
- модель токсичности в формате `TorchScript`.

Перед запуском модели текст проходит предобработку: убираются ссылки и упоминания, затем текст разбивается на токены и переводится в числовую форму через словарь. После этого модель вычисляет вероятность токсичного содержания.

Сервис не принимает окончательное решение сам. Он возвращает:

- статус;
- категории риска;
- числовую оценку токсичности.

Дальше основной сервер уже решает, что делать с материалом.

## Сервис проверки изображений

Сервис `image-filter` тоже запускается отдельно и вызывается сервером по `HTTP`.

В текущей реализации используется модель `CLIP`. Она не выбирает класс из длинного справочника, а сравнивает изображение с двумя текстовыми описаниями:

- фотография памятника;
- нерелевантное изображение или мусорный снимок.

Если изображение ближе к первому описанию, оно получает статус `accepted`. Если ближе ко второму — статус `garbage`.

Как и в случае с текстом, итоговое решение остается за прикладной логикой основного сервера.

## Локальный запуск

Развертывание выполняется в четыре шага: база данных, сервисы проверки, сервер, клиентская часть.

### 1. Подготовка базы данных

Нужна база PostgreSQL с подключенным PostGIS. После создания базы должно быть выполнено:

```sql
CREATE EXTENSION IF NOT EXISTS postgis;
```

### 2. Запуск сервиса проверки изображений

```powershell
cd C:\Users\iosir\OneDrive\Desktop\проект\image-filter
python -m venv venv
.\venv\Scripts\activate
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

### 3. Запуск сервиса проверки текста

```powershell
cd C:\Users\iosir\OneDrive\Desktop\проект\text-filter
python -m venv venv
.\venv\Scripts\activate
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 8001 --reload
```

### 4. Настройка и запуск серверной части

Файл `.env` в каталоге `project2/back` должен содержать как минимум:

```env
APP_PORT=8080
DATABASE_URL=postgresql://postgres:пароль@localhost:5432/имя_базы
JWT_SECRET=локальный_секрет
IMAGE_FILTER_URL=http://127.0.0.1:8000
TEXT_FILTER_URL=http://127.0.0.1:8001
UPLOADS_DIR=uploads
REGIONS_GEOJSON_PATH=AL2-AL8/russia_al2al8.geojson
EMAIL_ENABLED=false
```

После этого сервер запускается так:

```powershell
cd C:\Users\iosir\OneDrive\Desktop\проект\project2\back
go run ./cmd/app
```

При старте автоматически применяются миграции базы данных.

### 5. Запуск клиентской части

```powershell
cd C:\Users\iosir\OneDrive\Desktop\проект\project2\front
npm install
npm run dev
```

## Переменные окружения сервера

| Переменная | Назначение |
|---|---|
| `APP_PORT` | порт основного сервера |
| `DATABASE_URL` | строка подключения к PostgreSQL |
| `JWT_SECRET` | секрет для токенов доступа |
| `IMAGE_FILTER_URL` | адрес сервиса проверки изображений |
| `TEXT_FILTER_URL` | адрес сервиса проверки текста |
| `UPLOADS_DIR` | каталог локального хранения файлов |
| `REGIONS_GEOJSON_PATH` | путь к файлу регионов |
| `EMAIL_ENABLED` | включает или выключает отправку почты |
| `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`, `SMTP_FROM` | параметры почтового сервера |
| `EMAIL_WORKERS` | число фоновых обработчиков отправки почты |

## Хранение данных

В базе данных выделены основные сущности:

- пользователи и роли;
- точки памяти;
- посты и фотографии;
- сигналы и комментарии;
- жалобы и голоса по жалобам;
- уведомления;
- санкции;
- журнал событий;
- события уровня доверия.

Координаты и пространственные операции обслуживаются через PostGIS. Для определения региона используется файл `GeoJSON` с границами субъектов.

## Что важно учитывать

- новая точка может быть создана без первого поста;
- первый пост добавляется только по желанию пользователя;
- удаление точки работает по мягкому сценарию и не должно уничтожать чужие материалы без разбора;
- проверка текста и изображений является вспомогательной частью системы и не заменяет модерацию;
- локальное файловое хранилище используется для исходных изображений, превью и миниатюр.
